## 요약

README에 들어갈 런타임 입력의 핵심 사실은 확인됐다. 현재 프로젝트는 Spring Boot/Gradle 기반이고, 애플리케이션 실행 task는 `bootRun`, 테스트 task는 `test`/`check`다. `dev` profile은 PostgreSQL datasource 환경 변수와 JWT secret을 필요로 하며, Flyway baseline migration을 먼저 적용한 뒤 Hibernate `ddl-auto=validate`로 schema mapping을 검증한다.

현재 환경에서 `./gradlew test`는 성공했다. Gradle report 기준 275 tests, 0 failures, 0 ignored, 100% successful이다.

README 작성 전에 명확히 분리해야 할 setup 공백도 있다. 첫 MANAGER 계정을 만드는 bootstrap/seed 경로가 구현되어 있지 않다. 또한 `bootRun --args='--spring.profiles.active=dev'`는 현재 `main()`이 command-line args를 Spring Boot에 전달하지 않아 profile 활성화 방법으로 쓸 수 없다.

## Verified README input list

### Runtime and commands

- Project root command: `./gradlew bootRun`
- Verified application task: `bootRun - Runs this project as a Spring Boot application.`
- `dev` profile activation: `SPRING_PROFILES_ACTIVE=dev` 환경 변수로 동작 확인.
- Do not document `./gradlew bootRun --args='--spring.profiles.active=dev'` as the primary profile activation command until `main(args: Array<String>)` passes args to `runApplication`.
- Default HTTP port: Spring Boot default `8080`; no explicit `server.port` override found.

- Repository currently has no `docker-compose.yml`, `compose.yml`, Dockerfile, or local PostgreSQL provisioning script. This matches the completed Neon foundation ticket, which explicitly chose Neon for development/deployment instead of a local always-on Docker Compose database.

### Required runtime environment variables

| Variable | Required for local `dev` run | Default | Notes |

### Flyway and JPA validation

- Flyway dependencies are present: `org.flywaydb:flyway-core` and `org.flywaydb:flyway-database-postgresql`.
- Baseline migration file: `src/main/resources/db/migration/V1__create_mvp_schema.sql`.
- `dev` profile uses `spring.jpa.hibernate.ddl-auto=validate`.
- Spring Boot Flyway auto-configuration runs before JPA entity manager initialization.
- `MvpSchemaMigrationTest` verifies the baseline creates MVP tables, selected constraints, selected indexes, and `flyway_schema_history`.
- Integration tests that use the PostgreSQL Testcontainers JDBC URL generally set `spring.jpa.hibernate.ddl-auto=validate`, so README can say core DB-backed tests are migration-backed instead of Hibernate schema-generation-backed.

### PostgreSQL, Neon, and Testcontainers

- Project documents define PostgreSQL as the target DBMS, with Neon as the development/deployment PostgreSQL provider.
- `.env.example` is Neon-oriented and uses `jdbc:postgresql://...?...sslmode=require`.
- Automated integration tests use PostgreSQL Testcontainers via `jdbc:tc:postgresql:16-alpine:///test`.
- Testcontainers annotations use `@Testcontainers(disabledWithoutDocker = true)`, so Docker availability controls whether those tests can run.
- No local PostgreSQL container definition exists in the repository. This is consistent with the completed Neon foundation ticket; README should document the Neon/external PostgreSQL path and not imply Docker Compose support.

### Bootstrap and seed-account assumptions

- No `CommandLineRunner`, `ApplicationRunner`, `data.sql`, `schema.sql`, seed script, or migration-inserted admin account was found.
- `POST /api/v1/auth/login` is public, but `POST /api/v1/admin-users` requires `MANAGER`.
- Without an existing active MANAGER row, a fresh database has no documented way to create the first admin account through the API.
- Follow-up resolved in `docs/16-first-manager-bootstrap-path.md`: README should document the supported manual SQL bootstrap path, not a demo login or automatic seed account.

### F-04: `bootRun --args` profile activation is broken by the current Kotlin `main()`

- Category: `Bug`
- Source document reference: Notion `Audit README readiness inputs`
- Code or test reference: `src/main/kotlin/dev/climbdesk/ClimbDeskApplication.kt`
- Current behavior: `./gradlew bootRun --args='--spring.profiles.active=dev'` logs `No active profile set` because `main()` calls `runApplication<ClimbDeskApplication>()` without passing command-line args.
- Expected behavior: Either README should use environment variable/system-property profile activation, or `main(args: Array<String>)` should pass args to Spring Boot.
- Impact: A README using the common `bootRun --args=...` pattern would be misleading.
- Recommended action: Create a follow-up implementation ticket to pass command-line args, or document only `SPRING_PROFILES_ACTIVE=dev`.
- Risk and effort estimate: Low risk, low effort.

## 감사 결론

README에 필요한 실행 command, env var, Flyway/JPA, Testcontainers 사실은 검증됐다. M8 README는 Neon/external PostgreSQL 경로를 기준으로 작성하면 된다. 첫 MANAGER 계정 생성 경로는 `docs/16-first-manager-bootstrap-path.md`의 수동 SQL bootstrap 절차를 사용한다. `bootRun --args`는 현재 동작하지 않으므로 README에는 `SPRING_PROFILES_ACTIVE=dev` 기반 실행을 사용해야 한다.
